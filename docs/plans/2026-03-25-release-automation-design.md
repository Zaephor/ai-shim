# Release Automation Design

## Goal

Automate versioning, changelog generation, and release cutting using release-please and GoReleaser on a single protected `main` branch.

## Architecture

Trunk-based development on `main`. Contributors open PRs with conventional commits. Release-please maintains a standing Release PR that accumulates version bumps and changelog entries. Merging the Release PR tags `main`, which triggers GoReleaser to build binaries and create the GitHub Release.

## Workflow

```
Feature PR merged to main
        |
        +---> ci.yml (lint, test, e2e, build)
        |
        +---> release-please.yml
                |
                +---> Creates/updates Release PR
                        |
                        | (maintainer reviews + merges)
                        |
                        +---> release-please tags vX.Y.Z
                                |
                                +---> release.yml (GoReleaser)
                                        |
                                        +---> GitHub Release + binaries
```

## New Files

### `.github/workflows/release-please.yml`

Triggers on push to `main`. Runs the release-please action with Go release type. Uses `GITHUB_TOKEN` (no PAT needed).

### `release-please-config.json`

Central config defining the root package with Go release type. Changelog sections grouped by commit type: Features, Bug Fixes, Performance, Refactoring. Build, chore, docs, test, and CI commits hidden from changelog.

### `.release-please-manifest.json`

Version state file. Starts at `0.0.0` (no existing tags). Release-please updates this automatically on each release.

## Existing Files

No changes to any existing files:
- `ci.yml` -- unchanged
- `release.yml` -- unchanged
- `.goreleaser.yaml` -- unchanged
- `cmd/ai-shim/main.go` -- `version` stays `"dev"`, GoReleaser ldflags inject at build time

## Versioning

- Version determined entirely from conventional commits (feat = minor, fix = patch, breaking = major)
- No version file maintained in source -- git tags are the source of truth
- GoReleaser reads the tag via ldflags `-X main.version={{.Version}}`
- Release-please manages `.release-please-manifest.json` for its own state

## Changelog

Two changelogs coexist:
- `CHANGELOG.md` in repo -- cumulative history, owned by release-please
- GitHub Release notes -- per-release, generated by GoReleaser independently

Both group by commit type. Slight format differences are expected and acceptable.

## Branch Protection

Repository rulesets to configure when repo goes public or joins an org:
- Require pull request before merging (1 approval)
- Require status checks: lint, test, e2e, build
- Block force pushes
- Block branch deletion

## Decisions

- Release type: `go`
- Version tracking: git tags only (no version file)
- Changelog format: GoReleaser-style, grouped by commit type
- GoReleaser changelog: coexists independently with release-please CHANGELOG.md
- Release PR: requires review (deliberate release gate)
- Initial version: calculated by release-please from commit history (starts from 0.0.0 manifest)
